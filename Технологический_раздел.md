# 3. Технологический раздел

## 3.1. Выбор средств разработки
Обоснование выбора технологического стека производилось исходя из требований к производительности, наличия готовых библиотек для машинного обучения и сетевого взаимодействия:
- **Язык программирования**: Python 3.10+. Выбран благодаря богатой экосистеме библиотек для машинного обучения и работы с сетью, а также высокой скорости прототипирования.
- **Машинное обучение**: Библиотека `scikit-learn` (в частности, алгоритм Isolation Forest). Идеально подходит для задачи поиска аномалий без учителя. Отдельный анализ рядов производится с использованием структур `numpy` и `pandas`.
- **Сетевое взаимодействие**: Библиотека `paho-mqtt` для интеграции с протоколом MQTT (в том числе для подписки на топики Sparkplug B) и `protobuf` для декодирования бинарных структур данных полезной нагрузки.
- **Веб-интерфейс**: Микрофреймворк `Flask`. Обеспечивает легковесную реализацию легко расширяемого REST API и рендеринг HTML-шаблонов веб-дашборда без избыточной сложности (роутинг реализован в `src/web/app.py`).
- **База данных**: `SQLite`. Встраиваемая реляционная СУБД, не требующая развертывания отдельного сервера, идеально подходящая для хранения истории аномалий и сырого трафика (таблицы `anomalies`, `traffic`, `devices`, `tags`).

## 3.2. Архитектура системы
Система состоит из нескольких независимых, но взаимодействующих модулей, что обеспечивает гибкость и масштабируемость:
1. **Модуль захвата трафика (TrafficCapture)**: отвечает за подключение к MQTT-брокеру (например, `broker.hivemq.com` или локальному Mosquitto) и подписку на топики `spBv1.0/#`.
2. **Модуль извлечения признаков (FeatureExtractor)**: буферизует входящие сообщения по устройствам, вычисляет статистические метрики (среднее, стандартное отклонение, минимум, максимум, дисперсию интервалов) за заданное временное окно (например, 60 секунд).
3. **Модуль детектирования аномалий (AnomalyDetector)**: связующее ядро системы, объединяющее ML-модель с обработчиком топиков и парсером метрик.
4. **Модуль хранения (DatabaseManager)**: инкапсулирует SQL-запросы для сохранения сырого трафика, конфигурационных настроек и зафиксированных инцидентов-аномалий.
5. **Веб-дашборд (Flask App)**: предоставляет эндпоинты API и визуальный пользовательский интерфейс для мониторинга потока данных и просмотра журнала.

## 3.3. Реализация ключевых компонентов

Далее приведены основные фрагменты исходного кода, демонстрирующие техническую реализацию функционала.

### 3.3.1. Подключение к MQTT и захват трафика
Для получения данных из промышленной сети реализован класс `TrafficCapture` в модуле `mqtt_client.py`.

**Листинг 1 — Подключение клиента к MQTT-брокеру**
```python
    def connect(self) -> bool:
        try:
            logger.info(f"Подключение к {self.broker_host}:{self.broker_port}...")
            self.client.connect(self.broker_host, self.broker_port, 60)
            self.client.loop_start()
            
            # Ожидание подключения
            import time
            for _ in range(10):  # 5 секунд максимум
                if self.connected:
                    return True
                time.sleep(0.5)
            
            logger.error("Таймаут подключения")
            return False
            
        except Exception as e:

            return False
```
*Пояснение к Листингу 1:* Метод выполняет асинхронное подключение к заданному MQTT-брокеру с таймаутом в 5 секунд. При успешном соединении запускается фоновый цикл обработки сетевых пакетов (`loop_start`).

### 3.3.2. Парсинг бинарных сообщений Sparkplug B
Протокол Sparkplug B использует бинарный формат Protobuf. В системе предусмотрен двойной механизм декодирования: основной через сгенерированные классы Protobuf и резервный (fallback) ручной разбор байтовых последовательностей на случай поврежденных пакетов.

**Листинг 2 — Извлечение метрик из полезной нагрузки Sparkplug B**
```python
            # 1. Сначала пробуем как бинарный Sparkplug B (Protobuf)
            try:
                from .sparkplug_b_pb2 import Payload as SparkplugPayload
                inbound_payload = SparkplugPayload()
                inbound_payload.ParseFromString(msg.payload)
                for metric in inbound_payload.metrics:
                    val = None
                    if metric.HasField('int_value'): val = float(metric.int_value)
                    elif metric.HasField('long_value'): val = float(metric.long_value)
                    elif metric.HasField('float_value'): val = float(metric.float_value)
                    elif metric.HasField('double_value'): val = float(metric.double_value)
                    
                    if val is not None:
                        metrics_to_process.append({
                            "name": metric.name if metric.name else f"alias_{metric.alias}",
                            "value": val
                        })
            except Exception as e:
                # 2. Если Protobuf не сработал, используем ручной Fallback-декодер
                logger.debug(f"Protobuf decoder failed, using fallback: {e}")
                metrics_to_process = SparkplugManualDecoder.decode_metrics(msg.payload)
```
*Пояснение к Листингу 2:* Код из модуля `anomaly_detector.py` пытается десериализовать входящий `payload` с помощью схемы `sparkplug_b_pb2`. Значения метрик (целые числа или числа с плавающей точкой) приводятся к единому типу `float`. В случае ошибки парсинга вызывается эвристический алгоритм побайтового поиска.

### 3.3.4. Обучение модели и детектирование аномалий (Isolation Forest)
Основным аналитическим ядром системы является алгоритм Изолирующего леса (Isolation Forest), реализованный с использованием библиотеки `scikit-learn`. Данный алгоритм был выбран как оптимальный для задачи потокового поиска аномалий без учителя: он изолирует нетипичные наблюдения случайным разбиением признакового пространства и требует минимальных ресурсов для работы в реальном времени.

Модель инкапсулирована в классе `AnomalyDetector`. Инициализация модели производится со следующими параметрами: гиперпараметр `contamination` (ожидаемая доля выбросов в данных) по умолчанию равен $0.05$, количество деревьев `n_estimators` — $100$.

Архитектура классификатора предусматривает два основных режима работы:
1. **Режим обучения (Collect)**: Система собирает массив векторов эталонного поведения. Для обеспечения стабильного обучения алгоритма при недостатке входных данных (менее 50 сэмплов) алгоритм применяет аугментацию: исходные векторы клонируются с добавлением 1%-го нормального гауссовского шума. Это позволяет расширить тренировочный массив для создания корректных разделяющих деревьев в Isolation Forest. Перед обучением все признаки нормализуются с использованием `StandardScaler`. Обученная модель сохраняется на диск (модуль `pickle`) для быстрого перезапуска.
2. **Режим детектирования (Detect)**: Поступающий вектор признаков масштабируется и прогоняется через модель для получения метки состояния.

**Листинг 4 — Классификация состояния (Норма/Аномалия)**
```python
    def predict_single(self, features: np.ndarray) -> Tuple[int, float]:
        # Преобразуем в 2D массив
        X = features.reshape(1, -1)
        
        # Нормализация данных
        X_scaled = self.scaler.transform(X)
        
        # Предсказание метки состояния и степени аномальности
        labels = self.model.predict(X_scaled)
        scores = self.model.decision_function(X_scaled)
        
        if len(labels) > 0:
            return labels[0], scores[0]
        else:
            return 1, 0.0
```
*Пояснение к Листингу 4:* Вектор признаков нормализуется, после чего `IsolationForest` возвращает метку класса (`1` — норма, `-1` — аномалия) и сырую оценку (`score`) из метода `decision_function`, которая показывает удаленность экземпляра от разделяющей гиперплоскости (значение меньше нуля означает аномалию). 

Дополнительно в классе `AnomalyDetector` реализована нормализация этой оценки в вероятность выброса, на основе которой определяется уровень критичности:
- **Критичный инцидент (Critical)** при вероятности $> 0.6$
- **Высокий (High)** при вероятности $> 0.4$
- **Средний (Medium)** для пограничных случаев (вероятность $\le 0.4$, но метка равна $-1$)

Интеграция такой логики позволяет администратору в веб-интерфейсе фильтровать зафиксированные в SQLite инциденты по приоритету угрозы.

### 3.3.5. Веб-интерфейс
Пользовательский интерфейс реализован через HTTP-эндпоинты и REST API для динамического обновления данных.

**Листинг 5 — API получения сводной статистики для дашборда**
```python
@app.route('/api/summary')
def get_summary():
    """API: Получить сводную информацию для дашборда"""
    stats_24h = db.get_anomaly_statistics(hours=24)
    stats_1h = db.get_anomaly_statistics(hours=1)
    recent_anomalies = db.get_recent_anomalies(limit=5)
    devices = db.get_devices()
    recent_traffic = db.get_recent_traffic(limit=10)
    
    summary = {
        "anomalies_24h": stats_24h['total'],
        "anomalies_1h": stats_1h['total'],
        "devices_count": len(devices),
        "recent_anomalies": safe_serialize(recent_anomalies),
        "top_devices": safe_serialize(sorted(devices, key=lambda x: x['tags_count'], reverse=True)[:5]),
        "settings": {
            "training_period_minutes": db.get_setting('training_period_minutes', '60'),
            "system_mode": db.get_setting('system_mode', 'collect')
        }
    }
    return jsonify(summary)
```
*Пояснение к Листингу 5:* Сервер на базе Flask собирает агрегированные данные из SQLite (количество аномалий, активные устройства, текущий режим работы — `collect` или `detect`) и сериализует их в формате JSON для безопасного отображения на дашборде. Шаблоны страниц рендерятся методами `/`, `/traffic` и `/anomalies`.

## 3.4. Результаты тестирования
Работоспособность системы была проверена с помощью интеграционного и модульного тестирования (папка `tests/`). Настроены автоматизированные сценарии:
- `test_mqtt_client.py`: проверяет корректность подключения к брокерам, подписки на топики и срабатывания колбэков при получении пакетов из внешних источников (включая MQTT Lab).
- `test_binary_parsing.py`: валидирует корректность работы декодера Protobuf-сообщений при разборе сырых дампов трафика протокола Sparkplug B.
- `test_device_api.py`: проверяет стабильность веб-сервера Flask и консистентность JSON-ответов от REST эндпоинтов архитектуры системы.

Модель `IsolationForest` успешно тестировалась как на локальных тестовых выборках с добавлением шума (вспомогательные механизмы в модулях `extractor.py` и `model.py`), так и на синтетическом трафике инжектора. Система бесперебойно переходит из режима накопления эталонных данных в режим классификации в реальном времени. Во время локальных испытаний (отражено в `system.log`) алгоритм стабильно детектирует неестественные скачки значений датчиков, безошибочно помечая их как аномалии с указанием уровня угрозы в базе данных.
